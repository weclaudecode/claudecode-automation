Question 2 — How does HTML get its content?

You said it yourself: "we can use the same content to reduce duplication". This is the core trade-off — drift risk vs build complexity vs how fast HTML reflects what Claude just wrote to the MD.

A

Dual-write

Claude writes both MD + HTML
1
Claude edits ADR-008.md
↓
2
Claude also updates ADR-008.html
↓
3
You open the HTML
HTML reflects MD?If Claude remembers
Drift risk?High
Build step?None
Token cost / edit?2× content
+ Each artifact is fully self-contained HTML
− Every MD edit means re-writing HTML too
− Search/filter requires N HTML files to know about each other
"Skill" feel — but Claude becomes the build system.
B

Live render (Recommended)

HTML template loads MD at runtime
1
Claude edits ADR-008.md — that's it
↓
2
You run docs serve (or it's already running)
↓
3
Viewer fetch('ADR-008.md') → renders
HTML reflects MD?Instantly on refresh
Drift risk?Zero (single source)
Build step?None — server scans live
Token cost / edit?1× (MD only)
+ Single source of truth — MD is canonical
+ Search/filter via tiny /api/index the server builds live
+ Skill stays focused on writing MD well
− Needs docs serve running (one command)
"Automated flow" feel — template loads content. No drift, ever.
C

Static site build

mkdocs/eleventy-style
1
Claude edits ADR-008.md
↓
2
Claude runs docs build (post-edit hook)
↓
3
Pre-rendered ADR-008.html drops into _viewer/
HTML reflects MD?After build (~1-3s)
Drift risk?If build skipped
Build step?Required, every edit
Token cost / edit?1× (just MD)
+ Pure static — no server needed, deploys anywhere
+ Mature ecosystem (mkdocs-material is excellent)
− Build must run after every Claude edit (hook needed)
− Less interactive (search is pre-indexed lunr/typesense)
Industry-standard but optimised for "publish docs to web", not "private dev tool".
The recommendation is B because you said "reduce duplication" — the only model where MD is genuinely the single source of truth. C is reasonable if you'd rather avoid running a local server. A is the option I'd actively talk you out of — it puts Claude in the build-system loop forever.
