"Skill or automated flow?" — both, complementary

Originally you framed this as either/or. The architecture makes them complementary: the viewer renders any MD; a small skill teaches Claude to write MD that classifies correctly.
dev-artifact-viewer AUTOMATED FLOW
Astro site. Knows nothing about Claude. Reads MD + frontmatter, renders typed layouts, builds search index. Run via docs serve / docs build. Ships in the engine repo (next question).
dev-artifact-author SKILL
Tiny skill that triggers when Claude is about to write an ADR / plan / review. Provides the frontmatter schema + section conventions for each type. Output: MD that the viewer renders perfectly. No HTML generation.

Open question 2 — How does the viewer decide an MD is an ADR vs a Plan vs ...?

A
Frontmatter is authoritative · filename pattern is fallbackRECOMMENDED
If frontmatter has type: adr, use it. Otherwise infer from filename (ADR-NNN-*.md → adr, plans/*.md → plan, audits/*.md → review). Existing files work without modification; new files written by Claude will include frontmatter via the skill.
B
Frontmatter required
Build fails if any MD doesn't declare type:. Stricter, but means migrating every existing file. Loud failures, no ambiguity.
C
Filename pattern only
No frontmatter dependency. Pattern ADR-NNN-*, plans/*, etc. Simpler but you can't override the type for outliers, and ADR status comes from parsing prose (fragile).
